checkout, reset, revert
	- summary
		- checkout:	used when exploring history
		- reset:	used when reshaping local work
		- revert:	used when undoing shared history
	- checkout
		- moves HEAD
		- gives you a detached head, orphaning commits which come after
		  the checkout point
	- reset
		git reset <filename>
			- removes <filename> from staging area
		git reset
			- removes all files from the staging area
		git reset <commit>??
		- considered to be a moderately dangerous command as it orphans
		  commits (which are eventually garbage-collected)
		- moves a branch (current branch to new commit point?)
			--soft
				- branch only
				- staging area (index) remains unchanged
				- use this when you've made three commits and want to
				  combine them into one:
				  		reset --soft <commitpoint>; commit;
			--mixed (the default)
				- branch is moved and staging (index) is reset
				- changes still exist in files, just unstaged
				- you've committed, but want to restage it differently;
				  maybe split it into multiple commits
			--hard
				- everything is reset:
					- branch is moved to the commit point
					- staging area reset
					- working directory reset
				- files change; uncommitted work is gone from the disk
				- use this when you want to completely abandon work and
				  start fresh
	- revert (adds corrective history)
		- add a new commit that does the opposite of an old commit
			- eg.: one commit adds 50 lines of code, revert removes those
			  50 lines of code
		- what you're recording in history is that you've decided to
		  undo what you did earlier
			- you can't rewrite shared history, but you can add to it
		- use this when you need to undo something that's already been
		  pushed and shared

rebase (rewrites history)
	- NEVER REBASE COMMITS THAT YOU'VE ALREADY PUSHED
	- replays commits with new parents
		- the branch disappears and commits are applied sequentially to
		  the end of the branch to which you are rebasing
	- only rebase local, unpushed commits

when you've really fucked up you can "replay the reflog"
	- the reflog is the map to where HEAD has pointed recently
	- git branch recovery <commitID>
	- reflog entries expire after 90 days for reachable commits, and
	  30 days for orphaned commits
	- git reflog shows the recent history


Push Problems
-------------
If someone modifies a file you are working on, the push will fail.  If you use pull, a new merge+commit will be performed (because git pull is just a git fetch followed by a git merge).  If everyone does this then the commit history is going to look like the dog's dinner.  Better instead to just put your commit after the other person's commit:

	git pull --rebase

If there is a merge conflict then you can:

	git rebase --abort
	git pull

git pull --ff-only
git pull --ff
git pull --rebase
	- pull

Commit Best Practices
---------------------
- commit often
	- BUT: not too big, not to small
	- "real-world programmers commit 5-10 times per day, depending on the
	  kind of work you're doing" (JUST A GUILDLINE)
- each commit should reflect a logically separate transaction
	- one commit for the typo, one commit for the bugfix
	- this also makes it easier to make meaningful commit messages
- meaningful commit messages
	- they're going to show up in the commit history
	- need to be helpful to you and the other team members
	- this is much easier to do if each commit represents a logically
	  separate transaction
- message tense
	- some teams use the present tense ("Fix the bug"), other teams use
	  the past tense ("Fixed the bug")
	- whichever you use, stick to it

GIT GUI

Core Purpose & Focus

Unlike alternative tools that visualize complex branch trees or project histories, git-gui is a task-oriented application designed to help you craft precise commits. It allows you to selectively stage specific lines or hunks of code from a file, helping you avoid accidentally committing temporary debug statements or incomplete changes.


Key Features

- Commit Crafting: Easily view unstaged changes, move files to the staged
  area, type commit messages, and amend your last commit.
- Line & Hunk Staging: Right-click within the diff view to stage or unstage
  individual chunks or specific lines of text instead of the whole file.
- File Blaming & Browsing: Includes a built-in "blame" viewer to inspect
  file annotations and a basic file browser to explore past commits.
- Basic Repository Actions: Create new branches, perform local merges, and
  fetch, push, or prune remote repositories.
- History Integration: While it doesn't display project history natively,
  it provides quick menu shortcuts to launch gitk (Git's companion history
  viewing tool).


The Interface Layout

When you launch the tool by typing git gui into your terminal, the interface is split into four primary quadrants:
	1. Top Left: Unstaged changes (files you have modified but haven't
	   added to the index).
	2. Bottom Left: Staged changes (files ready to be committed).
	3. Top Right: The diff view, showing the precise line modifications of
	   the selected file.
	4. Bottom Right: The commit message box and action buttons (Commit,
	   Push, Amend).


How to Use It

Because it comes bundled with the core software, you do not need to install a heavy third-party client. To use it, open your terminal or Git Bash, navigate to your project folder, and run: git gui 

https://git-scm.com/docs/git-gui


GITK

gitk is a built-in, graphical repository browser for Git that displays a visual map of a project's commit history. Written in Tcl/Tk, it historically stands as the very first repository browser created alongside Git. It provides an lightweight alternative to running complex command-line arguments like git log --graph.


Key Features

- Commit Graph Visualizer: Displays a dynamic tree structure showing how
  branches diverge, merge, and connect over time.
- Diff Viewer: Selecting any commit instantly reveals what lines were
  added (green) or removed (red) across modified files.
- Deep History Search: Allows filtering and searching through commits by
  specific keywords, authors, SHA-1 hashes, or even a specific line of
  code.
- Contextual Actions: Right-clicking branches or commits opens shortcuts
  to cherry-pick changes, reset branches, or create new tags.


Basic Usage

To use gitk, open your terminal, navigate to any local Git repository directory, and run:  gitk


Common Command Options

You can filter the history graph upon launch using arguments directly supported by the gitk Documentation:

	- gitk --all: View the commit history of every local and remote branch
	  simultaneously.
	- gitk --since="2 weeks ago": Restrict the visual graph to recent
	  changes.
	- gitk <branch_name>: Isolate and explore the timeline of one specific
	  branch.


Distinction from git-gui

While gitk is bundled with Git and targets history visualization, it is often confused with its sibling tool git-gui. gitk acts as a viewer to read repository history, while git-gui is an interactive tool built for authoring changes, staging files, and creating new commits


Sign-Off
--------
From Gemini:

In Git, a "sign-off" is a standard line appended to the end of a commit message certifying that the author has the legal right to submit the code under the project's open-source license. It typically looks like this: Signed-off-by: Your Name <your.email@example.com>.


Purpose and Legal Meaning

The sign-off serves as a digital agreement to the Developer Certificate of Origin (DCO). It was originally created by the Linux Foundation to establish a clear legal chain of custody for code contributions. By signing off, you affirm that:
	- You wrote the code yourself, or based it on a compatible open-source
	  license.
	- You have the right to contribute it to the project.


How to Use Sign-Off

You can apply a sign-off to your work using these Git commands:

	- New Commit: Add -s or --signoff to your commit statement to append
	  the signature line using your globally configured name and email.
			git commit -s -m "Your commit message"
	- Existing Commit: Add a sign-off line to your most recent commit.
	 		git commit --amend --signoff
	- During a Rebase: Apply a sign-off automatically to a series of
	  commits.
			git rebase --signoff main


Sign-Off vs. Commit Signing

These two Git terms are often confused but serve entirely different functions:

Feature		Git Sign-Off				Commit Signing
Format		Plain text appended to the	Cryptographic digital signature
			commit message				(GPG/SSH keys)
Purpose		Legal confirmation of code	Identity verification to prevent
			ownership and licensing		author spoofing
Git Flag	-s or --signoff				-S or --gpg-sign
Visibility	Public text readable inside	Green "Verified" badge on
			any git log					platforms like GitHub
